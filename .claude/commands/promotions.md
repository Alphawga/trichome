# Promotions Reference

Use this skill when touching admin promotions/discounts (`src/app/admin/promotions/`), the `Promotion`/`PromotionUsage` models, `src/server/modules/promotions.ts` + `src/server/modules/promotions-storefront.ts`, `src/lib/promotions/*`, or the site's top announcement banner (`src/components/layout/header.tsx`).

**Changed 2026-07-17**: added location scoping (`applicable_state`/`applicable_city`) and a `display_location` flag (`CHECKOUT`/`PRODUCT_TAG`/`BOTH`), and rewrote resolution from "pick exactly one promotion" to "stack every eligible codeless promotion + one optional manual code on top" — several facts below that were true before that date are corrected inline.

## No cron — status is self-healed lazily on read

There is no job scheduler anywhere in this app. `Promotion.status` (`ACTIVE`/`INACTIVE`/`SCHEDULED`/`EXPIRED`) does not transition on its own just because a date passed — a `SCHEDULED` promo whose `start_date` arrives, or an `ACTIVE` promo whose `end_date` passes, stays in that stale status forever unless something touches it. `syncPromotionStatuses()` in `src/server/modules/promotions.ts` fixes this: it runs two `updateMany` calls (`SCHEDULED`→`ACTIVE` where `start_date <= now`, `ACTIVE`→`EXPIRED` where `end_date < now`) and is called at the top of every read procedure (`getPromotions`, `getPromotionById`, `getPromotionStats`, `getBannerPromotions`). **If you add a new read path over `Promotion`, call `syncPromotionStatuses(ctx.prisma)` at the top of it too** — the DB is never migrated to a correct state, it's repaired on the next read.

## Promotions are store-wide across products — but can be location-scoped

The `Promotion` model still has no `product_ids`/`category_ids` field — discounts apply to every product uniformly, gated only by `min_order_value`, `target_customers` (`ALL`/`NEW_CUSTOMERS`/`VIP`/`SPECIFIC_GROUP`), and usage limits. If a task asks for "20% off skincare products only," that still requires a schema change, not just a new promotion row.

It CAN be geographically scoped, though: `applicable_state`/`applicable_city` (both nullable strings, added 2026-07-17) let an admin restrict a promotion to a delivery destination (e.g. the old hardcoded "free shipping for Akure only" rule, now a real admin-editable `FREE_SHIPPING` promotion with `applicable_state: "Ondo", applicable_city: "Akure"`). `checkPromotionEligibility` (`src/lib/promotions/promotion-eligibility.ts`) takes optional `state`/`city` and rejects (case/whitespace-insensitive) on a mismatch; missing `state`/`city` (address not yet known) also fails the match rather than erroring, so a location-scoped promo just doesn't appear until the destination is known. City requires state to be set too (enforced server-side in `createPromotion`/`updatePromotion`). A location-scoped promotion can't also be `display_location: PRODUCT_TAG`/`BOTH` (see below) — enforced both client-side (`PromotionDiscountFields.tsx` disables those options, `PromotionFormSheet.tsx`'s `useEffect` force-resets to `CHECKOUT`) and server-side, since the product-tag read has no destination context to check against.

`applicable_state` should use `NIGERIAN_STATES` (`src/lib/constants/nigerian-states.ts`) values, not free text — matches the convention already used by `calculate-shipping.ts`/`AddressForm.tsx`.

## `validatePromoCode` re-checks dates independently of `status`

`validatePromoCode` (`promotions.ts`, public procedure used at checkout) checks `promotion.status !== "ACTIVE"` **and separately** checks `now < start_date || now > end_date`. This means even before the `syncPromotionStatuses` fix existed, a stale `ACTIVE` promo past its `end_date` could never actually be applied at checkout — the status bug was a display/admin-UX problem (misleading `SCHEDULED` promos, wrong-looking stats), not a way to get an expired discount to apply.

## Site banner is driven by a single flag, not a general CMS

`Header` (`src/components/layout/header.tsx`, client component) queries the public `getBannerPromotions` tRPC procedure, which returns **every** `Promotion` matching `show_on_banner: true && status === "ACTIVE" && start_date <= now <= end_date` (empty array if none) — changed 2026-07-17 from a `findFirst`/singular `getBannerPromotion` that silently dropped all but the newest flagged promo when two or more were active simultaneously. The evergreen "NATURAL BEAUTY, NATURALLY YOURS - SHOP NOW" tagline is a permanent first item; every active promotion is appended after it, each rendered as `name` + optional ` - description` + optional ` - USE CODE {code}` + a "LEARN MORE" link, all separated by `•`. Desktop and mobile now render identically (the old mobile-only "swap tagline for the single promo" behavior is gone). When there are 2+ items the row scrolls in a continuous seamless-loop marquee (`.animate-marquee` in `globals.css`, duration scales with item count) instead of the old `overflow-hidden whitespace-nowrap` clipping; with 0 active promotions it's just the static centered tagline, unchanged from before. There is still no generic reusable banner/announcement-bar component in this codebase; this is the only DB-driven text in that slot. **The banner still applies no location scoping** — unlike checkout/product-tag resolution, a promotion restricted to `applicable_state`/`applicable_city` shows in the banner regardless of visitor location; this is a known, not-yet-addressed gap.

The marquee's looping duplicate copy is `aria-hidden` **and** `inert` (React 19 prop) — `aria-hidden` alone isn't enough, since it removes content from the accessibility tree but doesn't affect Tab order, so a screen-reader-invisible copy of the "LEARN MORE"/"SHOP NOW" links would otherwise still be keyboard-focusable. Any future duplicated-for-animation content in this codebase should follow the same `aria-hidden` + `inert` pairing, not `aria-hidden` alone.

## Admin form conventions

`PromotionFormSheet.tsx` uses `react-hook-form` with `reset()` for all three modes (edit/template-prefill/blank-create) — any new field needs a default in all three `reset()` calls or it'll be `undefined` on some paths. Boolean fields use a plain `<input type="checkbox" {...register("field")} />`, matching the pattern in `ProductFormSheet.tsx` (no dedicated Checkbox/Switch UI component exists in `src/components/ui/`). The file was split into `PromotionFormSheet.tsx` + `PromotionDiscountFields.tsx` on 2026-07-07, then `PromotionLocationFields.tsx` was split out too on 2026-07-17, all to stay under this repo's ~600-line convention — put discount-type-specific fields in `PromotionDiscountFields.tsx`, location-targeting fields in `PromotionLocationFields.tsx`, schedule/targeting/banner/code fields in the form-shell file.

There's a `has_code` checkbox (added 2026-07-17, `PromotionFormSheet.tsx`) positioned above the `code` input — it's **form-only state**, not a DB field; unchecking it hides the code input and forces submission as codeless regardless of stale leftover text in the field. Any code checked out of `useForm`'s values via `{ has_code, ...rest }` before building the mutation payload.

**Numeric fields need `setValueAs`, not bare `register()`.** `react-hook-form`'s `register()` on an `<input type="number">` returns the raw DOM string unless told otherwise — `value`, `min_order_value`, `max_discount`, `usage_limit`, `usage_limit_per_user` all hit this (fixed 2026-07-16: submitting a typed-in number, as opposed to only dragging the paired `Slider`, sent a string to `createPromotion`'s `z.number()` Zod schema and threw `invalid_type`). Don't reach for `valueAsNumber: true` here — on an emptied field it produces `NaN`, not `undefined`/`0`, which fails Zod's `z.number()` in a more confusing way than a normal "required" error. Use an explicit `setValueAs: (v) => (v === "" ? <default> : Number(v))` per field instead: `undefined` for genuinely optional fields (`max_discount`, `usage_limit_per_user`) so they stay `.optional()`-compatible, `undefined` for required fields too (so RHF's own `required` rule still fires a proper message), `0` for `min_order_value` (its real semantic default). Any new numeric field on this form needs the same treatment — the Slider's own `setValue(field, values[0])` calls are unaffected (they already pass real numbers, bypassing `register` entirely), only the paired text `<Input>` needs it.

## `code` is optional (since 2026-07-15) — codeless means auto-apply, and ALL eligible codeless promotions stack

`Promotion.code` is `String?` (nullable, still `@unique` — Postgres allows multiple `NULL`s under a unique index). **No code is the "automatic" signal** — there's no separate flag (`has_code` on the admin form is UI-only, see above). `createPromotion`/`updatePromotion` only run the code-uniqueness `findUnique`/`findFirst` check when a code is actually provided; `updatePromotion`'s Zod schema distinguishes omitting `code` (leave unchanged) from sending `code: null` (explicitly clear it back to codeless) via `z.union([z.string()..., z.null()]).optional()`. UI call sites that assume `promotion.code` is always a string (copy-to-clipboard buttons, banner "USE CODE X" text, CSV/table cells) all need a `promotion.code ? ... : <fallback>` guard — `PromotionViewSheet.tsx`, `admin/promotions/page.tsx`, and `header.tsx`'s banner text already do this; check any *new* UI surface that reads `.code` too.

**As of 2026-07-17, resolution changed from "pick exactly one promotion" to "stack every eligible codeless promotion, plus one optional manual code on top."** The old `resolveOrderPromotion`/`getAutoApplyPromotion` (singular) were replaced:
- `getEligibleCodelessPromotions` + `resolveOrderPromotions` (plural, `src/lib/promotions/promotion-eligibility.ts`) — the shared "fetch all `code: null`, `ACTIVE`, in-range, `min_order_value`-satisfied candidates, filter by `checkPromotionEligibility`" step lives in `getEligibleCodelessPromotions`; `resolveOrderPromotions` folds that plus an optional coded promotion into `{ promotions: ResolvedOrderPromotion[], totalDiscountAmount, isFreeShipping }`. Each promotion's discount is computed **independently off the original subtotal and summed** (additive, not sequentially compounded — no ordering dependency between promotions), then capped at the subtotal once at the end. `isFreeShipping` is an OR across the whole stack.
- `getAutoApplyPromotions` (plural, `promotions-storefront.ts`) — public procedure, calls `getEligibleCodelessPromotions` and returns the array (used to be `getAutoApplyPromotion`, returning the single newest-eligible candidate or `null`).
- `orders.ts`'s two order-creation procedures dropped their `promotion_id` input entirely — the server now derives the whole automatic stack itself from `subtotal`/`state`/`city`, so a client can no longer identify "this is the one auto-applied promotion" by id (closing that trust surface, not just relocating it). Only `promo_code` remains client-supplied. `PromotionUsage` write-back is now a loop, one row + one `usage_count` increment **per stacked promotion actually applied**, not one row with the summed total.
- `CheckoutClient.tsx`: `activePromo` (singular) → `autoAppliedPromos` (array, from the query) + `appliedPromoCode` (still singular — only one manually typed code can be applied at a time, and codes never stack with each other). The order summary lists every automatic promotion by name/amount plus the manual code on its own line, instead of one "Promo applied" line.
- `getActiveProductTagPromotions` (new, `promotions-storefront.ts`) reuses the same additive-stacking rule for product-tag price display (`calculateProductTagDiscount` in `src/lib/promotions/product-tag-discount.ts`) — consistent between what a product card shows and what checkout actually charges for store-wide percentage/fixed promos.

## Product price-tag display (`display_location`, added 2026-07-17)

`Promotion.display_location` (`CHECKOUT`/`PRODUCT_TAG`/`BOTH`, default `CHECKOUT`) lets an admin flag a `PERCENTAGE`/`FIXED_AMOUNT` codeless promotion to show as a struck-through price on every product card/PDP, store-wide (still no product/category scoping, so it discounts every product uniformly). `canDisplayOnProductTag(type)` (`src/lib/promotions/promotion-display-rules.ts`) is the single source of truth for "is this discount type eligible" — called from `PromotionDiscountFields.tsx` (client gating), `PromotionFormSheet.tsx`'s force-reset `useEffect`, and both `createPromotion`/`updatePromotion` server validation; don't reimplement the `type === "PERCENTAGE" || type === "FIXED_AMOUNT"` check inline anywhere new, import the helper.

`getActiveProductTagPromotions` (public, `promotions-storefront.ts`) has no cart/subtotal/address context, so `min_order_value` is **intentionally ignored** for tag-display eligibility — it calls `checkPromotionEligibility(..., { subtotal: Number.MAX_SAFE_INTEGER, ... })` so that check always passes regardless of what `min_order_value` an admin set (fixed 2026-07-17, same day as the feature shipped: it originally passed `subtotal: 0`, which does the *opposite* — `checkPromotionEligibility` rejects whenever `subtotal < min_order_value`, so `0` silently excluded every promotion with a nonzero minimum, including the very first one a client created with it. If you touch this procedure again, don't "fix" it back to `0` or any real subtotal — the whole point is this check must always pass here). Location restriction is still enforced normally (which is why location-scoped + `PRODUCT_TAG`/`BOTH` together is rejected at save time, see above — that one's a real incompatibility, not a numeric-comparison bug). Returns raw promo terms (`type`, `value`, `max_discount`), not a precomputed price — the client computes the discounted price per-product via `calculateProductTagDiscount` (`src/lib/promotions/product-tag-discount.ts`), called from `product-card.tsx` and `ProductDetailsClient.tsx`. When a tag promotion is active it **fully replaces** (not merges with) the product's manually-set `compare_price` strikethrough — a product decision, confirmed explicitly, not "whichever is lower." `product-card.tsx` calls `trpc.getActiveProductTagPromotions.useQuery({})` directly (not threaded down as a prop from `ProductsPageClient.tsx`) — React Query dedupes the identical query key across every rendered card into one network request, so this is intentional, not an N+1.

## Eligibility/discount logic lives outside `src/server/modules/` on purpose

`checkPromotionEligibility`/`calculatePromotionDiscount`/`getEligibleCodelessPromotions`/`resolveOrderPromotions` are **not** in `promotions.ts` — they're in `src/lib/promotions/promotion-eligibility.ts` (`syncPromotionStatuses` similarly lives in `src/lib/promotions/sync-promotion-statuses.ts`, shared by both `promotions.ts` and `promotions-storefront.ts`). This isn't a style choice: every file under `src/server/modules/*.ts` is spread wholesale into the tRPC router via `import * as xModule from "./modules/x"` + `...xModule` in `src/server/index.ts` (see Architecture section of the repo's root CLAUDE.md). A plain exported function (not a `publicProcedure`/`adminProcedure`/etc. chain) inside one of those files gets mistaken for a router entry and **breaks the router's type for the entire app** — every `appRouter.createCaller(...)` call site fails with a cryptic `DecorateRouterRecord<any>` / "This expression is not callable" error, including in unrelated test files. If you need to extract shared business logic (not a procedure) out of a `server/modules/*.ts` file, it must go in `src/lib/`, matching the existing `src/lib/shipping/calculate-shipping.ts`/`src/lib/payments/calculate-paystack-fee.ts` convention — never leave a bare exported helper in a module file that's spread into the router. `promotions.ts` (admin/staff procedures) and `promotions-storefront.ts` (public procedures) were split on 2026-07-17 to stay under the ~600-line file convention — both are spread into `appRouter` in `src/server/index.ts`.

## Guest checkout previously silently ignored `promo_code` (fixed 2026-07-15)

`createGuestOrderWithPayment` (`src/server/modules/orders.ts`) accepted a `promo_code` input field and destructured it but never used it anywhere — guest checkout applied zero server-side promotion validation and trusted whatever `totals.discount` the client sent. Fixed by porting in the same promotion-resolution logic `createOrderWithPayment` already had. One deliberate, permanent limitation: `PromotionUsage.user_id` is non-nullable and guest orders have no `user_id`, so `usage_limit_per_user` can never be enforced for guests — only the global `usage_limit` applies (via `Promotion.usage_count` increment, which still happens for guests). Don't try to "fix" this without a schema change to make `PromotionUsage.user_id` nullable, which wasn't in scope for this pass.

## FREE_SHIPPING promotions need the server to actually zero shipping

`computeServerShippingCost` doesn't know about promotions — it always returns the real computed rate. If a `FREE_SHIPPING`-type promotion resolves as eligible, the caller must skip calling it (`isFreeShipping ? 0 : await computeServerShippingCost(...)`) rather than calling it and discarding the result — calling it anyway both wastes an external shipping-rate API call and, if the caller then ignores its result, produces server logic that visually looks like it accounts for shipping but doesn't.
