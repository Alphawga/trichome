# Shipbubble Shipping Reference

Use this skill when touching shipping-rate quoting, checkout shipping method selection, or the Shipbubble adapter ā€” `src/lib/shipping/get-shipping-rate.ts`, `src/lib/shipping/providers/*`, `src/server/modules/shipping.ts`, the shipping-related sections of `src/server/modules/orders.ts` (`computeServerShippingCost`), `src/app/(customer)/checkout/CheckoutClient.tsx`.

## Architecture (fallback-first by design)

`getShippingRates()` (`src/lib/shipping/get-shipping-rate.ts`) is the single entry point, called both by the checkout-quote tRPC procedure and by order creation. It tries the live Shipbubble adapter only if `SHIPBUBBLE_API_KEY` is set, and falls back to the static per-state table (`src/lib/shipping/calculate-shipping.ts`, unmodified) on any error, timeout, missing key, or **incomplete destination** (missing `addressLine`/`contactName`/`contactEmail`/`contactPhone`). The incomplete-destination case logs a distinct `console.warn` ("skipping live quote... not a Shipbubble outage") instead of the generic `console.error` used for real operational failures, so the two are greppably distinguishable. The completeness check itself lives in one place ā€” `hasCompleteDestinationForLiveQuote()` in `shipbubble-provider.ts` ā€” reused by both the pre-check in `get-shipping-rate.ts` and `validateReceiverAddress`'s own defensive throw, so they can't drift apart.

**2026-07-03 fix**: `CheckoutClient.tsx`'s shipping-rate query used to fire (and silently fall back to static) as soon as state+city were known, before phone/address/email were typed ā€” this was the root cause of a "shipment API isn't working, still shows the static base rate" bug report. The `enabled` gate now requires all 4 Shipbubble-required fields (debounced together, 500ms) before firing, the UI shows "Complete your name, email, phone number and street address..." instead of a number until then, and a static-fallback rate that does render is tagged "(estimated)" (`ShippingRate.source === "static"`). The "Continue to Payment" button is also now disabled while the debounce/query hasn't caught up with the latest typed input (`isShippingQuotePending`) ā€” this closes a real payment-amount-mismatch risk: `computeServerShippingCost` recomputes shipping server-side from the just-submitted address at order-creation time, and a customer submitting before the debounced quote settles could have their Paystack charge diverge from the server-recomputed total. Note `computeServerShippingCost`'s own address schema (`orders.ts`) still allows `phone` optional ā€” the checkout gate doesn't protect that path, so it's still possible to reach the incomplete-destination fallback via order creation directly (not through checkout's UI). If shipping numbers ever look wrong or suspiciously "static," check whether the live path is even configured, and check server logs for which of the two distinct warn/error paths fired, before assuming a bug in the adapter.

## Never trust client-reported shipping cost (the trust boundary)

Same pattern as Paystack payment verification (see `paystack-payments` skill): `createOrderWithPayment`/`createGuestOrderWithPayment` (`src/server/modules/orders.ts`) do **not** persist the client-supplied `totals.shipping`. They call `computeServerShippingCost()`, which recomputes shipping authoritatively server-side by calling `getShippingRates()` again and using `rates[0].cost` directly ā€” the client-reported figure is never consulted. A fraudulent/zeroed client value simply fails the subsequent Paystack amount-mismatch check (compared against the server-recomputed total, not the client's).

`getShippingRates()` always returns exactly one rate (`ShippingRate[]` with a single element) ā€” there is no standard/express or multi-courier choice surfaced to the customer. Checkout shows that one computed rate as a read-only line, not a picker.

## Shipbubble API contract (confirmed against their live docs, not assumed)

- **Auth**: `Authorization: Bearer <SHIPBUBBLE_API_KEY>`. Key prefix tells the mode: `sb_sandbox_...` (test) vs `sb_prod_...` (live, real wallet debits). Never hardcode a key literal ā€” `edit-check.sh` flags `"sb_prod_..."`/`"sb_sandbox_..."` string literals same as it does Paystack keys.
- **A quote is two sequential calls, not one**:
  1. `POST /v1/shipping/address/validate` ā€” body `{ name, email, phone, address }` (free-text address string; `latitude`/`longitude` optional and take priority if given). Returns a normalized `address_code` (number) plus normalized city/state/postal/lat-lon. Required for **both** sender and receiver ā€” there's no way to pass a raw address straight into the rate endpoint. **`name` must be a two-word personal name** (e.g. `"John Doe"`) ā€” a single-word brand name like `"Trichomes"` gets rejected with a 422 (`"Please provide a full name (e.g John Doe), please remove all numbers and symbols"`). Use a real contact person's name, not the business name, when validating the sender/warehouse address.
  2. `POST /v1/shipping/fetch_rates` ā€” body needs `sender_address_code`, `reciever_address_code` (Shipbubble's own spelling, not a typo to fix), `pickup_date` (`yyyy-mm-dd`), `category_id`, `package_items` (`{ name, description, unit_weight, unit_amount, quantity }[]`), `package_dimension` (`{ length, width, height }` cm ā€” **required**, no default in their API). Optional `service_type` (`dropoff`/`pickup`).
- **Response**: `data.couriers[]`, each with `courier_name`, `total` (wallet-debit amount if booked), **`rate_card_amount`** (the customer-facing price ā€” use this one, not `total`), a `discount` object (`percentage`/`discounted`, not currently surfaced in checkout UI), `pickup_eta_time`/`delivery_eta_time`. `shipbubble-provider.ts` only reads `data.cheapest_courier` (falling back to `couriers[0]`) ā€” `fastest_courier` is ignored, since checkout no longer offers a method choice. Also a `request_token` (valid 7 days, only needed if booking a shipment later ā€” not used yet, quoting only).
- **`pickup_date` constraints**: must be within 7 days of the request date ā€” max lookahead, requests further out are rejected. Requests made after 6PM GMT+1 get auto-scheduled by Shipbubble for the next day regardless of the `pickup_date` sent, so `nextBusinessDay()` (`shipbubble-provider.ts`) doesn't need to special-case time-of-day itself.
- **`category_id` and the sender `address_code` are one-time, account-specific setup values**, not computed at request time: `category_id` comes from `GET /v1/shipping/labels/categories` (merchant-specific list ā€” this account's confirmed live list, 2026-07-02: Hot food, Dry food and supplements, Electronics and gadgets, Groceries, Sensitive items (ATM cards, documents), Light weight items, Machinery, Medical supplies, **Health and beauty (`99652979`, used for Trichomes)**, Furniture and fittings, Fashion wears ā€” no generic "Cosmetics" entry, "Health and beauty" is the closest fit), and the sender `address_code` comes from validating the warehouse address once (Trichomes' Akure address from `site-config.ts` ā†’ `569748124`, validated under contact name "Nonso Chillington" per the name-format rule above). Both live in `.env` as `SHIPBUBBLE_CATEGORY_ID`/`SHIPBUBBLE_SENDER_ADDRESS_CODE` ā€” if either is missing/wrong, every live quote fails and silently falls back to static (check these before assuming a code bug).

## Known gaps (as of 2026-07-03)

- `Product` has no L/W/H dimension fields ā€” `shipbubble-provider.ts` uses a fixed generic small-parcel `package_dimension` (20Ć—20Ć—15cm) for every quote, regardless of actual product size.
- No DB column distinguishes live vs. static-fallback shipping quotes on an `Order` ā€” only logged via `console.warn`/`console.error` on fallback (see the two distinct paths above). Add a column if this needs querying/auditing later.
- `get-shipping-rate.test.ts` covers the static-fallback path and `getShippingRates()`'s live/incomplete/operational-failure branching (mocked `fetch`), but not the real HTTP contract in `shipbubble-provider.ts` (address validation + fetch_rates request/response shapes) or `computeServerShippingCost` in `orders.ts` ā€” still untested end-to-end.
- `computeServerShippingCost`'s address schema (`orders.ts`) still has `phone` as `.optional()`, unlike checkout's now-required gate ā€” order creation can still hit the incomplete-destination static fallback if some future caller omits phone, independent of the checkout fix.
- **Live and confirmed end-to-end as of 2026-07-02** ā€” `SHIPBUBBLE_CATEGORY_ID`/`SHIPBUBBLE_SENDER_ADDRESS_CODE` are set and a real courier quote (not static fallback) was observed in checkout. The one open item: the `SHIPBUBBLE_API_KEY` in `.env` was flagged for rotation (previously exposed in chat) but never actually rotated ā€” confirm with the user before assuming it's safe long-term. See `docs/changelog/2026-07-02-shipbubble-shipping-integration.md` and the `project-shipbubble-integration` memory for full session history.
