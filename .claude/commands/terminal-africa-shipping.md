# Terminal Africa Shipping Reference

Use this skill when touching shipping-rate quoting, checkout shipping method selection, or the Terminal Africa adapter — `src/lib/shipping/get-shipping-rate.ts`, `src/lib/shipping/providers/*`, `src/server/modules/shipping.ts`, the shipping-related sections of `src/server/modules/orders.ts` (`computeServerShippingCost`), `src/app/(customer)/checkout/CheckoutClient.tsx`.

## Architecture (fallback-first by design)

`getShippingRates()` (`src/lib/shipping/get-shipping-rate.ts`) is the single entry point, called both by the checkout-quote tRPC procedure and by order creation. It tries the live Terminal Africa adapter (`terminal-africa-provider.ts`) only if `TERMINAL_SECRET_KEY` is set, and falls back to the static per-state table (`src/lib/shipping/calculate-shipping.ts`, unmodified) on any error, timeout, missing key, or **incomplete destination** (missing `addressLine`/`contactName`/`contactEmail`/`contactPhone`). The incomplete-destination case logs a distinct `console.warn` ("skipping live quote... not a Terminal Africa outage") instead of the generic `console.error` used for real operational failures, so the two are greppably distinguishable. The completeness check lives in one place — `hasCompleteDestinationForLiveQuote()` in `terminal-africa-provider.ts` — reused by both the pre-check in `get-shipping-rate.ts` and the provider's own defensive throw.

**2026-07-06: replaced Shipbubble entirely.** Shipbubble's account for this store hit a confirmed, reproducible `"Insufficient wallet balance"` error on rate quoting (see `docs/changelog/2026-07-02-shipbubble-shipping-integration.md` and the retired provider's history for context) — the live path was silently falling back to static rates with no staff-visible alert. Terminal Africa replaces it as the only live provider; the static fallback and its debounce/gating behavior in `CheckoutClient.tsx` are unchanged from the Shipbubble era.

## Never trust client-reported shipping cost (the trust boundary)

Same pattern as Paystack payment verification (see `paystack-payments` skill): `createOrderWithPayment`/`createGuestOrderWithPayment` (`src/server/modules/orders.ts`) do **not** persist the client-supplied `totals.shipping`. They call `computeServerShippingCost()`, which recomputes shipping authoritatively server-side by calling `getShippingRates()` again and using `rates[0].cost` directly. `getShippingRates()` always returns exactly one rate — no standard/express or multi-courier choice surfaced to the customer.

## Terminal Africa API contract — **UNVERIFIED against a real response as of 2026-07-06**

The account's KYC has been submitted and is pending review; a live test call 401'd with `"Complete KYC verification to access the API"` before any request-shape question could be resolved. Everything below is built from Terminal Africa's public docs (via a GitBook AI doc-summarizer that gave at least one internal contradiction), not a confirmed live call. **Re-verify all of this with a real request once KYC clears**, and update this section with "confirmed live" once done — don't assume it's correct just because it's documented here, the same way Shipbubble's real contract (one-time setup codes, wallet debits) only became clear via live testing despite having docs too.

- **Auth**: `Authorization: Bearer <TERMINAL_SECRET_KEY>`. Base URL: sandbox `https://sandbox.terminal.africa/v1`, production `https://api.terminal.africa/v1` (`TERMINAL_AFRICA_BASE_URL` env override, defaults to sandbox). `TERMINAL_PUBLIC_KEY` also exists in `.env` but isn't used by this server-side integration — unclear purpose, don't invent one.
- **Rate quote — believed to be one call**: `POST /rates/shipment/quotes`, embedding full `address_from`/`address_to` objects directly (`first_name`/`last_name`/`line1`/`city`/`state`/`country`/`zip`/`email`/`phone`) plus `parcel_total_weight`, `parcel_value`, `packages: [{ weight, length, width, height }]`, `currency: "NGN"`. `carrier_id` is optional — omitting it is supposed to return quotes from multiple carriers (aggregator mode).
  - **Open contradiction**: a separate doc query claimed you must create pickup/delivery addresses first via `POST /addresses` (returns `data.address_id`) and reference them by ID instead of embedding them. `terminal-africa-provider.ts` is built against the embedded-object version. **If the first real sandbox call 400s/401s (once KYC clears) on a field-shape error, add a `createAddress()` step and switch to ID references** — see the code comment above `fetchQuotes()` in that file for the exact fallback plan.
  - Unlike Shipbubble, there's no persisted one-time "sender address ID" or "category ID" — the sender/warehouse address is built inline from `site-config.ts` on every request, and there's no confirmed category/product-type field in the contract at all.
- **Response**: array of rate objects, each with `amount` (cost), `carrier_name`, `carrier_rate_description`, `delivery_eta` (**in minutes** — not a human string like Shipbubble's, converted to days via `Math.ceil(minutes / 1440)` in `estimateDaysFromEtaMinutes()`), `id`/`rate_id`. `terminalAfricaProvider.getRates()` picks the minimum by `amount` itself (no confirmed `cheapest`-flagged field in the response, unlike Shipbubble's `cheapest_courier`).
- **KYC is a compliance gate, not a cost gate** — confirmed via Terminal Africa's own "How to Complete KYC" guide (one-time BVN + valid-ID upload, no fee mentioned) and their published API pricing page, which lists exactly three paid services (Duty Calculation $0.04/call, Global Address Validation $0.02/call, HS Code Search $0.04/call) — rate quoting isn't on that list. Don't assume a 401/KYC gate implies a paywall, and don't assume it doesn't either — check each provider's own pricing docs (this is the opposite lesson from Shipbubble, whose gate looked like it might be "just KYC" but turned out to be a real wallet-balance cost).
- A `terminal-africa-tship-documentation` MCP server (GitBook-hosted, `~/.claude.json`, user scope) exists for querying live docs directly — prefer it over re-deriving the contract from a WebFetch doc-summary once it's connected (needs a Claude Code restart after being added).

## Known gaps (as of 2026-07-06)

- **The entire live contract above is unverified** — no successful real API call has been made yet (KYC pending review). This is the single biggest open item; do not report the live path as "working" until a real quote comes back with a named carrier and a non-round `amount` (the same static-vs-live tell used to debug Shipbubble: a suspiciously round number matching `STATE_SHIPPING_COSTS` means the fallback fired, not the live provider).
- `resolvedState` (`ShippingRate.resolvedState`) is now permanently unpopulated — Terminal Africa's rate response doesn't echo a geocoded address the way Shipbubble's `address/validate` step did. The field is kept optional in `types.ts` for a hypothetical future provider; the checkout UI hint that used to display it was removed (`CheckoutClient.tsx`).
- `Product` has no L/W/H dimension fields — `terminal-africa-provider.ts` uses a fixed generic small-parcel `package_dimension` (20×20×15cm) for every quote, same limitation as the retired Shipbubble adapter.
- No DB column distinguishes live vs. static-fallback shipping quotes on an `Order` — only logged via `console.warn`/`console.error` on fallback.
- `computeServerShippingCost`'s address schema (`orders.ts`) still has `phone` as `.optional()`, unlike checkout's required-fields gate — order creation can still hit the incomplete-destination static fallback if some future caller omits phone.
- `get-shipping-rate.test.ts` covers the static-fallback path and `getShippingRates()`'s live/incomplete/operational-failure branching (mocked `fetch`), but not the real HTTP contract — same testing gap the Shipbubble integration had until it was live-verified.
- **Not yet confirmed**: whether `TERMINAL_SECRET_KEY` has been added to the production environment (Vercel), not just local `.env` — needed for the live path to ever activate in production once KYC clears.

## Code-review findings (2026-07-06, confirmed, not yet fixed — user chose to defer)

- **`TERMINAL_AFRICA_BASE_URL` defaults to the sandbox host with nothing overriding it in `.env`.** If `TERMINAL_SECRET_KEY` turns out to be a production-issued key, every live request goes to `sandbox.terminal.africa` with a prod key, gets rejected, and silently falls back to static — indistinguishable in logs from the already-expected KYC-pending 401. **Check/set `TERMINAL_AFRICA_BASE_URL` explicitly before assuming KYC clearing alone will make live quotes work.**
- **`senderAddress()` uses the business name "Trichomes Cosmeceuticals"** split into `first_name`/`last_name` — the exact pattern that got Shipbubble's sender address rejected (422, personal name required) before the team switched to a real contact name. If Terminal Africa applies similar validation, this will repeat that failure once KYC clears, indistinguishable from a KYC-pending 401.
- **Cheapest-carrier selection (`rates.reduce`, comparing `.amount` with `<`) has no runtime check that `amount` is actually numeric.** A string-typed `amount` in the real response would make this a lexicographic comparison, silently picking the wrong (more expensive) carrier with no error.
- **`fetchQuotes()` never sends `input.items`** (the cart's per-item manifest) — the retired Shipbubble provider explicitly mapped items into `package_items`; this provider omits it, which could fail validation or produce an inaccurate quote if Terminal Africa's endpoint needs itemized data (plausible given it also does duty/customs calculation).
- **`receiverAddress()` sends `destination.state` verbatim** (full names like "Lagos"/"Abuja") with no normalization — unconfirmed whether Terminal Africa expects full names, abbreviations, or ISO 3166-2:NG codes.

**All five of the above should be re-checked and likely fixed once KYC clears and the first live call either succeeds or fails** — several of them would currently be indistinguishable from a KYC-pending 401 in logs, so don't assume "still 401ing" means "still waiting on KYC" without checking these first.
