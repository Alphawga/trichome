# Terminal Africa Shipping Reference

Use this skill when touching shipping-rate quoting, checkout shipping method selection, or the Terminal Africa adapter — `src/lib/shipping/get-shipping-rate.ts`, `src/lib/shipping/providers/*`, `src/server/modules/shipping.ts`, the shipping-related sections of `src/server/modules/orders.ts` (`computeServerShippingCost`), `src/app/(customer)/checkout/CheckoutClient.tsx`.

## Architecture (fallback-first by design)

`getShippingRates()` (`src/lib/shipping/get-shipping-rate.ts`) is the single entry point, called both by the checkout-quote tRPC procedure and by order creation. It tries the live Terminal Africa adapter (`terminal-africa-provider.ts`) only if `TERMINAL_SECRET_KEY` is set, and falls back to the static per-state table (`src/lib/shipping/calculate-shipping.ts`, unmodified) on any error, timeout, missing key, or **incomplete destination** (missing `addressLine`/`contactName`/`contactEmail`/`contactPhone`). The incomplete-destination case logs a distinct `console.warn` ("skipping live quote... not a Terminal Africa outage") instead of the generic `console.error` used for real operational failures, so the two are greppably distinguishable. The completeness check lives in one place — `hasCompleteDestinationForLiveQuote()` in `terminal-africa-provider.ts` — reused by both the pre-check in `get-shipping-rate.ts` and the provider's own defensive throw.

**2026-07-06: replaced Shipbubble entirely.** Shipbubble's account for this store hit a confirmed, reproducible `"Insufficient wallet balance"` error on rate quoting (see `docs/changelog/2026-07-02-shipbubble-shipping-integration.md` and the retired provider's history for context) — the live path was silently falling back to static rates with no staff-visible alert. Terminal Africa replaces it as the only live provider; the static fallback and its debounce/gating behavior in `CheckoutClient.tsx` are unchanged from the Shipbubble era.

## Never trust client-reported shipping cost (the trust boundary)

Same pattern as Paystack payment verification (see `paystack-payments` skill): `createOrderWithPayment`/`createGuestOrderWithPayment` (`src/server/modules/orders.ts`) do **not** persist the client-supplied `totals.shipping`. They call `computeServerShippingCost()`, which recomputes shipping authoritatively server-side by calling `getShippingRates()` again and using `rates[0].cost` directly. `getShippingRates()` always returns exactly one rate — no standard/express or multi-courier choice surfaced to the customer.

## Terminal Africa API contract — **CONFIRMED LIVE 2026-07-07**

KYC cleared, base URL fixed, and a real end-to-end live quote succeeded through the actual `getShippingRates()` code path: `{ courier: "DHL Express", cost: 5189.75, estimatedDays: 5, source: "live" }` — a real carrier and non-round amount, not the static fallback.

- **Auth**: `Authorization: Bearer <TERMINAL_SECRET_KEY>`. Base URL: sandbox `https://sandbox.terminal.africa/v1`, production `https://api.terminal.africa/v1` (`TERMINAL_AFRICA_BASE_URL` env override, defaults to sandbox — **now set in local `.env`**; Vercel production still needs the same value added separately, not yet done). `TERMINAL_PUBLIC_KEY` also exists in `.env` but isn't used by this server-side integration — unclear purpose, don't invent one.
- **Rate quote, confirmed contract**: `POST /rates/shipment/quotes` with **`pickup_address`/`delivery_address`** objects (`first_name`/`last_name`/`line1`/`line2`/`city`/`state`/`country`/`zip`/`email`/`phone`) — **not** `address_from`/`address_to` as originally built from a doc-summary contradiction — plus a single **`parcel`** object: `{ description, weight_unit: "kg", items: [{ name, description, currency, value (total for that line), quantity, weight (per unit, kg), type: "parcel" }] }`. `packaging_id`/`packaging` is correctly omitted — only required when `persist_data: true` (reusing the `rate_id` to actually book a shipment later, which this quote-only flow doesn't do); Terminal applies a default packaging automatically. `carrier_id` is optional — omitting it returns quotes from multiple carriers, and `terminalAfricaProvider.getRates()` picks the cheapest via `.amount`.
- **`line1` has a hard 45-character limit**, confirmed via `POST /addresses/validate`. Hits **both** addresses independently:
  - Sender: the real full warehouse address in `SITE_ADDRESS.streetAddress` (~130 chars, landmark-heavy) is too long — `SITE_ADDRESS.shippingLine1`/`shippingLine2` in `site-config.ts` hold a manually-curated short form of the *same physical address* for carrier use only; `streetAddress` is untouched and still used for the footer/structured-data (full address, customer/SEO-facing).
  - Delivery: customer-entered `destination.addressLine` is free text and regularly exceeds 45 chars (confirmed live with a real customer test address, 48 chars — "20 Federal site Abule Odu bustop. Egbeda, Lagos"). Since this can't be hand-curated per customer, `receiverAddress()` calls `splitAddressLine()` (`terminal-africa-provider.ts`) which auto-splits at the last word boundary at-or-before 45 chars into `line1`/`line2`. **Any code path that builds a delivery/pickup address for Terminal Africa must go through this splitter** — a raw `line1: someLongString` will silently 400 and fall back to static, easy to miss since the error message ("Delivery Address - Address line 1 must have a maximum of 45 characters") only shows up in server logs, not customer-facing.
- **Response**: array of rate objects, each with `amount` (cost), `carrier_name`, `carrier_rate_description`, `delivery_eta` (**in minutes**, converted to days via `Math.ceil(minutes / 1440)` in `estimateDaysFromEtaMinutes()`), plus a large amount of carrier-scoring metadata not currently used.
- **KYC is a compliance gate, not a cost gate** — confirmed via Terminal Africa's own "How to Complete KYC" guide (one-time BVN + valid-ID upload, no fee mentioned) and their published API pricing page, which lists exactly three paid services (Duty Calculation $0.04/call, Global Address Validation $0.02/call, HS Code Search $0.04/call) — rate quoting isn't on that list, and the confirmed-working live test above didn't error on cost.
- A `terminal-africa-tship-documentation` MCP server (GitBook-hosted, `~/.claude.json`, user scope) exists for querying live docs directly — used to resolve the `address_from`/`pickup_address` contradiction definitively; prefer it over re-deriving the contract from a WebFetch doc-summary.

## Known gaps (as of 2026-07-07)

- **Vercel production env var not yet set.** `TERMINAL_AFRICA_BASE_URL=https://api.terminal.africa/v1` is only in local `.env` — production will still hit the sandbox default (and fail, since the key is a live key) until this is added in Vercel. User chose to handle this themselves.
- `resolvedState` (`ShippingRate.resolvedState`) is permanently unpopulated — Terminal Africa's rate response doesn't echo a geocoded address the way Shipbubble's `address/validate` step did. The field is kept optional in `types.ts` for a hypothetical future provider; the checkout UI hint that used to display it was removed (`CheckoutClient.tsx`).
- No DB column distinguishes live vs. static-fallback shipping quotes on an `Order` — only logged via `console.warn`/`console.error` on fallback.
- `computeServerShippingCost`'s address schema (`orders.ts`) still has `phone` as `.optional()`, unlike checkout's required-fields gate — order creation can still hit the incomplete-destination static fallback if some future caller omits phone.
- `get-shipping-rate.test.ts` covers the static-fallback path and `getShippingRates()`'s live/incomplete/operational-failure branching (mocked `fetch`), not the real HTTP contract — the contract itself is now verified live (see above), but the test suite doesn't assert on request-body shape, so a future accidental revert of `pickup_address`/`delivery_address`/`parcel` wouldn't be caught by tests.
- **Cheapest-carrier selection (`rates.reduce`, comparing `.amount` with `<`) has no runtime check that `amount` is actually numeric** — confirmed numeric in the one live response seen so far, but still no defensive check.
