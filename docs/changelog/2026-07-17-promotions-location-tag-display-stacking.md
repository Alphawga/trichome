# 2026-07-17 — Location-scoped promotions, product price-tag display, and promotion stacking

## What changed

Three related changes to the Promotion system, requested together by the client:

1. **Location-scoped promotions** — replaces the removed hardcoded "free shipping over ₦30,000 for Akure" rule in `calculate-shipping.ts` with a real, admin-editable `Promotion`. Added `applicable_state`/`applicable_city` (nullable strings, migration `20260717070738_promotion_location_and_display_location`) so an admin can restrict any promotion (not just `FREE_SHIPPING`) to a delivery destination. `checkPromotionEligibility` (`src/lib/promotions/promotion-eligibility.ts`) now takes optional `state`/`city` and rejects on mismatch (case/whitespace-insensitive, matching the removed hardcode's comparison style); missing address just fails the match rather than erroring, so a location-scoped promo doesn't show until the customer's destination is known. `NIGERIAN_STATES`-backed `<Select>` in the new `PromotionLocationFields.tsx`.
2. **Product price-tag display** — new `Promotion.display_location` (`CHECKOUT`/`PRODUCT_TAG`/`BOTH`, default `CHECKOUT`) lets an admin flag a `PERCENTAGE`/`FIXED_AMOUNT` codeless promotion to show as a struck-through original price + discounted price on every product card/PDP, store-wide (still no product/category scoping). New public `getActiveProductTagPromotions` procedure + shared `calculateProductTagDiscount` helper (`src/lib/promotions/product-tag-discount.ts`), wired into `product-card.tsx` and `ProductDetailsClient.tsx`. Fully replaces (not merges with) a product's manually-set `compare_price` strikethrough while active.
3. **Promotion stacking** — codeless ("automatic") promotions now all combine at checkout instead of only the newest one applying; a manually typed code adds on top of that stack instead of replacing it. Rewrote `resolveOrderPromotion` (singular) → `resolveOrderPromotions` (plural, additive, capped at subtotal), `getAutoApplyPromotion` → `getAutoApplyPromotions`. `orders.ts`'s two order-creation procedures dropped their `promotion_id` input entirely (the server now derives the whole automatic stack itself), and `PromotionUsage` write-back is now one row per stacked promotion instead of one row with a summed total.
4. **Admin UX**: explicit `has_code` checkbox above the code field (form-only state — unchecking always means codeless/auto-apply, replacing the previous "leave blank" implicit convention).

Also split `src/server/modules/promotions.ts` into admin/staff procedures + `promotions-storefront.ts` (public procedures), and `PromotionFormSheet.tsx` further into `PromotionLocationFields.tsx`, to stay under the repo's ~600-line file convention.
